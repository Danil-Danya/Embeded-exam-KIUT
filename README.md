# Документация проекта встроенного сервера

## Введение

Данный проект представляет собой серверную часть учебной системы мониторинга, предназначенной для совместной работы программного обеспечения и электронного устройства. Сервер выполняет две основные задачи. С одной стороны, он ведёт учёт студентов, которые имеют доступ к интерфейсу системы. С другой стороны, он принимает данные от устройств, сохраняет результаты измерений и формирует сводную информацию для дальнейшего анализа.

Приложение написано на Node.js с использованием Express. Для работы с базой данных применяется Sequelize, а сама база реализована на SQLite. Такое решение удобно для учебной и лабораторной среды, поскольку не требует отдельного сервера базы данных и разворачивается в пределах одного каталога проекта.

Практический смысл системы состоит в следующем. Электронное устройство передаёт измерения на сервер по HTTP, а пользовательская часть получает сведения о зарегистрированных устройствах, истории измерений и текущем состоянии стенда. Таким образом, сервер выступает в роли центрального узла обмена данными между аппаратной и программной частью.

## Назначение и состав системы

Система включает в себя три логические сущности. Первая сущность описывает студента, то есть пользователя, который может пройти авторизацию, просматривать список зарегистрированных объектов, открывать собственный профиль и работать с дашбордом. Вторая сущность описывает устройство, подключённое к лабораторной установке. Третья сущность хранит результаты измерений, поступающих от устройства.

Архитектурно сервер состоит из одного файла приложения [index.js](e:/programing/учеба/embeded%20server/index.js), файла описания Swagger [swagger-definition.js](e:/programing/учеба/embeded%20server/swagger-definition.js), локальной базы данных `database.sqlite`, каталога `static/images` для фотографий студентов и конфигурационных файлов окружения. Документация Swagger доступна по адресу `http://127.0.0.1:3000/server-api/docs/swagger`.

## Технологическая основа

В проекте используются Express для маршрутизации HTTP-запросов, `jsonwebtoken` для выдачи токенов доступа, `bcrypt` для хэширования паролей студентов, `multer` для приёма фотографии в формате `multipart/form-data`, а также `swagger-jsdoc` и `swagger-ui-express` для формирования и публикации интерактивной документации API.

База данных SQLite размещается непосредственно в файле проекта. Это означает, что перенос сервера на другой компьютер не требует настройки отдельной СУБД: достаточно перенести каталог и сохранить корректный файл окружения.

## Конфигурация окружения

Основные параметры задаются в файле `.env`. В текущем варианте проекта используются следующие переменные.

| Переменная | Назначение |
| --- | --- |
| `PORT` | TCP-порт, на котором запускается HTTP-сервер |
| `HOST` | сетевой адрес сервера в локальной сети или на стенде |
| `DB_DIALECT` | используемый тип базы данных; в данном проекте это `sqlite` |
| `DB_STORAGE` | путь к файлу SQLite |
| `JWT_SECRET_ACCESS_KEY` | секретная строка для подписи JWT-токенов |

Пример минимальной конфигурации:

```env
PORT=3000
HOST=127.0.0.1

DB_DIALECT=sqlite
DB_STORAGE=./database.sqlite

JWT_SECRET_ACCESS_KEY=your_secret_key
```

## Запуск проекта

Для запуска достаточно установить зависимости и запустить сервер в режиме разработки. Последовательность действий выглядит следующим образом.

```bash
npm install
npm run dev
```

После запуска сервер становится доступен по адресу `http://127.0.0.1:3000`, а маршруты API располагаются под префиксом `/server-api/v1`. Документация Swagger открывается отдельно по адресу `http://127.0.0.1:3000/server-api/docs/swagger`.

## Структура данных

### Таблица студентов

Таблица студентов содержит сведения о пользователях, имеющих доступ к системе.

| Поле | Тип | Смысл |
| --- | --- | --- |
| `id` | integer | уникальный идентификатор студента |
| `first_name` | string | имя студента |
| `last_name` | string | фамилия студента |
| `password` | string | хэш пароля, сформированный сервером |
| `photo` | string | относительный путь к фотографии |
| `role` | string | роль пользователя в системе |
| `about_work` | string | краткое текстовое описание направления работы |

Фотография физически сохраняется в каталоге `static/images`, а в поле `photo` записывается путь вида `/static/images/<имя_файла>`.

### Таблица устройств

Таблица устройств описывает каждый аппаратный узел, который может взаимодействовать с сервером.

| Поле | Тип | Смысл |
| --- | --- | --- |
| `id` | integer | уникальный идентификатор устройства |
| `name` | string | название устройства |
| `serial_number` | string | серийный номер устройства |
| `is_active` | boolean | признак активности устройства |
| `created_at` | datetime | дата создания записи |
| `updated_at` | datetime | дата последнего изменения записи |

При создании нового устройства признак `is_active` устанавливается в значение `false`. После поступления первого измерения сервер переводит устройство в активное состояние.

### Таблица измерений ветра

Таблица измерений хранит результаты, поступающие от аппаратной части.

| Поле | Тип | Смысл |
| --- | --- | --- |
| `id` | integer | идентификатор измерения |
| `device_id` | integer | ссылка на устройство |
| `value` | integer | сырое значение датчика |
| `rpm` | integer | число оборотов в минуту |
| `created_at` | datetime | момент сохранения |
| `updated_at` | datetime | момент последнего обновления |

Связь между сущностями построена по принципу «одно устройство - много измерений». Это позволяет хранить и анализировать историю значений для каждого конкретного устройства.

## Общие правила работы API

Все маршруты прикладного интерфейса начинаются с префикса `/server-api/v1`. Обмен данными, как правило, выполняется в формате JSON. Исключением является создание студента, где запрос отправляется в формате `multipart/form-data`, поскольку вместе с текстовыми полями передаётся файл фотографии.

Авторизация реализована на основе JWT. После успешного входа сервер выдаёт токен вида Bearer. Этот токен должен передаваться в заголовке

```http
Authorization: Bearer <token>
```

Для запросов от электронного устройства важно учитывать два обстоятельства. Во-первых, перед началом передачи измерений устройство должно пройти авторизацию по серийному номеру. Во-вторых, при отправке измерения требуется не только токен в заголовке, но и поле `device_id` в теле запроса, поскольку именно по нему сервер связывает измерение с конкретной записью в базе.

Параметры `page` и `limit` для методов получения списков следует передавать обязательно. На основании этих параметров сервер вычисляет смещение и формирует пагинированный ответ.

## Полный перечень эндпоинтов

Ниже приведена сводная таблица маршрутов, после которой каждый из них рассмотрен отдельно.

| Метод | Маршрут | Назначение |
| --- | --- | --- |
| `POST` | `/device/auth` | авторизация устройства |
| `POST` | `/student/auth` | авторизация студента |
| `POST` | `/student` | создание студента с фотографией |
| `POST` | `/device` | создание устройства |
| `POST` | `/wind-measurement` | сохранение измерения ветра |
| `GET` | `/student/profile` | получение профиля авторизованного студента |
| `GET` | `/student` | получение списка студентов |
| `GET` | `/student/:id` | получение одного студента |
| `GET` | `/device` | получение списка устройств |
| `GET` | `/device/:id` | получение одного устройства |
| `GET` | `/device/:deviceId/chart` | получение данных для графика устройства |
| `GET` | `/dashboard` | получение сводных данных дашборда |

## Подробное описание маршрутов

### 1. Авторизация устройства

Маршрут `POST /server-api/v1/device/auth` предназначен для аппаратной части. Устройство передаёт свой серийный номер, а сервер проверяет, зарегистрировано ли оно в базе данных. При успешной проверке возвращается JWT-токен доступа.

Тело запроса:

```json
{
  "serial_number": "SN-001"
}
```

Успешный ответ:

```json
{
  "accessToken": "jwt_token"
}
```

Данный маршрут следует использовать первым в рабочем цикле микроконтроллера или внешнего вычислительного узла. Без него дальнейшая передача измерений будет отклонена защитным middleware.

### 2. Авторизация студента

Маршрут `POST /server-api/v1/student/auth` используется пользовательской частью. Для входа студент передаёт имя, фамилию и пароль. Сервер находит соответствующую запись, сравнивает пароль с сохранённым хэшем и при успехе возвращает токен доступа.

Тело запроса:

```json
{
  "first_name": "Данил",
  "last_name": "Сабитов",
  "password": "123456"
}
```

Успешный ответ:

```json
{
  "accessToken": "jwt_token"
}
```

Токен, выданный студенту, требуется для всех методов чтения, связанных со студентами, устройствами и дашбордом.

### 3. Создание студента

Маршрут `POST /server-api/v1/student` принимает данные нового студента и фотографию. Запрос должен иметь формат `multipart/form-data`, а имя поля для файла должно быть строго равно `photo`.

Текстовые поля запроса:

| Поле | Назначение |
| --- | --- |
| `first_name` | имя студента |
| `last_name` | фамилия студента |
| `password` | пароль в открытом виде |
| `role` | роль в системе |
| `about_work` | краткое описание деятельности |
| `photo` | файл фотографии |

Пример формы:

```bash
curl -X POST http://127.0.0.1:3000/server-api/v1/student \
  -F "first_name=Данил" \
  -F "last_name=Сабитов" \
  -F "password=123456" \
  -F "role=student" \
  -F "about_work=Работает с микроконтроллерами" \
  -F "photo=@C:/photos/avatar.jpg"
```

В ответе сервер возвращает созданный объект студента. Пароль в базе хранится не в открытом виде, а в форме хэша `bcrypt`.

### 4. Создание устройства

Маршрут `POST /server-api/v1/device` регистрирует новое устройство в системе. При создании сервер автоматически выставляет `is_active: false`.

Тело запроса:

```json
{
  "name": "Датчик ветра №1",
  "serial_number": "SN-001"
}
```

Успешный ответ содержит созданную запись устройства, включая её числовой идентификатор. Этот идентификатор необходимо сохранить, так как он используется в измерениях в поле `device_id`.

### 5. Сохранение измерения ветра

Маршрут `POST /server-api/v1/wind-measurement` предназначен для аппаратной части и защищён device-token авторизацией. Помимо заголовка `Authorization`, запрос должен содержать значения `device_id`, `value` и `rpm`.

Пример запроса:

```http
POST /server-api/v1/wind-measurement HTTP/1.1
Host: 127.0.0.1:3000
Authorization: Bearer <device_token>
Content-Type: application/json
```

```json
{
  "device_id": 1,
  "value": 220,
  "rpm": 1450
}
```

Если устройство найдено, сервер сохраняет измерение. Если до этого устройство имело статус `is_active: false`, оно автоматически переводится в активное состояние. Для электронщика это означает, что корректная работа канала обмена сразу отражается в базе данных и на дашборде.

### 6. Получение профиля авторизованного студента

Маршрут `GET /server-api/v1/student/profile` возвращает запись текущего пользователя, определяя её по токену доступа. Этот метод удобен при запуске пользовательского интерфейса, когда после входа необходимо получить персональные сведения без отдельного указания идентификатора.

### 7. Получение списка студентов

Маршрут `GET /server-api/v1/student?page=1&limit=10` возвращает пагинированный список студентов. Ответ содержит два поля: `count`, где хранится общее количество записей, и `rows`, где расположен массив объектов текущей страницы.

Данный метод защищён student-token авторизацией. Без корректного токена сервер вернёт ответ со статусом `401`.

### 8. Получение одного студента

Маршрут `GET /server-api/v1/student/:id` используется для просмотра отдельной карточки студента. В параметре пути передаётся целочисленный идентификатор записи.

Пример:

```bash
curl -H "Authorization: Bearer <student_token>" \
  http://127.0.0.1:3000/server-api/v1/student/1
```

### 9. Получение списка устройств

Маршрут `GET /server-api/v1/device?page=1&limit=10` возвращает пагинированный список устройств. По структуре ответ аналогичен списку студентов: сервер возвращает `count` и `rows`.

Данный метод полезен как для веб-интерфейса, так и для проверки состояния лабораторного стенда. С его помощью можно быстро определить, какие устройства зарегистрированы и какие из них перешли в активный режим.

### 10. Получение одного устройства

Маршрут `GET /server-api/v1/device/:id` возвращает карточку отдельного устройства по идентификатору. Метод защищён student-token авторизацией и предназначен для операторской части системы.

### 11. Получение данных графика устройства

Маршрут `GET /server-api/v1/device/:deviceId/chart` формирует данные, пригодные для построения графика во фронтенде. Сервер находит устройство, извлекает до тридцати последних измерений и формирует ответ из двух частей: `device` и `chart`.

Поле `chart.labels` содержит временные подписи, а `chart.datasets` включает два массива: значения RPM и сырые значения датчика. Для учебной работы такой формат удобен тем, что не требует дополнительной обработки на стороне клиента.

### 12. Получение данных дашборда

Маршрут `GET /server-api/v1/dashboard` возвращает сводную статистику по системе. В ответ входят общее число студентов, устройств, измерений, число активных устройств, а также текущие, максимальные и средние значения RPM. Кроме того, сервер возвращает последнее сохранённое измерение.

Этот маршрут особенно важен для демонстрации работы стенда на защите, поскольку позволяет быстро показать текущее состояние системы в одном запросе.

## Практический порядок работы со стендом

С точки зрения аппаратной части последовательность взаимодействия с сервером должна быть следующей. Сначала устройство один раз проходит авторизацию по своему серийному номеру и получает токен. Затем при каждом цикле измерения оно отправляет на сервер JSON-пакет, содержащий `device_id`, `value` и `rpm`, обязательно сопровождая запрос заголовком `Authorization: Bearer <token>`.

С точки зрения пользовательской части работа строится иначе. Студент проходит авторизацию, получает собственный токен, после чего может просматривать список студентов, список устройств, карточки отдельных записей, профиль, дашборд и график по конкретному устройству.

## Форматы ответов

Сервер использует достаточно простую структуру ответов. В случае успешного создания записи обычно возвращается сам созданный объект. В случае получения списка возвращается объект с двумя полями: `count` и `rows`. В случае авторизации сервер возвращает объект с единственным полем `accessToken`.

При ошибках сервер формирует JSON-объект, в котором обязательно присутствует поле `message`. Внутренние ошибки дополнительно могут сопровождаться полем `error`, содержащим технические сведения.

## Документация Swagger

Для проверки маршрутов вручную в проект встроена Swagger-документация. Она расположена по адресу:

```text
http://127.0.0.1:3000/server-api/docs/swagger
```

Swagger полезен не только для программной части, но и для лабораторной отладки. Через него можно быстро проверить корректность тел запросов, наличие обязательных полей и общий формат ответов сервера.

## Заключение

Представленный сервер решает типичную для учебного проекта задачу связующего звена между аппаратной платформой и пользовательским интерфейсом. Он объединяет хранение данных, авторизацию, приём измерений и предоставление аналитической информации в одном приложении. Благодаря использованию SQLite, Express и Swagger проект остаётся достаточно простым для развёртывания, но при этом охватывает полный цикл обмена данными, необходимый для демонстрации работы встроенной системы в университетской среде.
